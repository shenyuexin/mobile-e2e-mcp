# 分阶段验收与验证策略（中文版）

本文档定义每个阶段的“完成标准”、验证方法、证据要求，以及你当前本地环境（Xcode + Android Studio 模拟器）的验收建议。

## 0. 验收总原则

1. **每阶段必须可量化**：不是“能跑”，而是“达到阈值”。
2. **每阶段必须可复现**：同一环境多次运行结果稳定。
3. **每阶段必须留证据**：失败必须有完整证据包。
4. **先模拟器后真机**：先稳定主链路，再扩展复杂场景。

---

## 1. 通用验证结构（每个 Phase 都用）

每个阶段必须输出以下内容：

- 验收范围（In Scope / Out of Scope）
- 验收环境（OS、模拟器型号、App 版本、配置）
- 用例清单（核心链路 + 边界链路）
- 验收指标（通过率、时延、失败分类）
- 证据包（截图、Tree、日志、Action Timeline）
- 审核结论（Go / No-Go）

建议补充“证据体积预算”字段（避免产物失控）：

- 单次 run 证据包目标体积上限（例如 5~15MB，按环境调整）
- 截图默认压缩档位（分辨率、格式、质量）
- 原图保留策略（全量/抽样/仅失败）

建议复用模板：

- `templates/acceptance-evidence-template.md`
- `templates/phase-review-checklist.md`
- `templates/sample-app-matrix-template.md`

## 2. Phase 0 验收（基础能力）

### 验收目标

- MCP 工具协议、错误码、会话模型、策略模型可用。

### 必达指标

- 关键工具 schema 版本化完成（100%）
- 错误码分类可覆盖主要失败路径（>= 90%）
- 会话开始/结束/中断恢复流程通过（>= 95%）

### 证据

- schema 文档
- 会话流程日志
- 错误样例映射表

---

## 3. Phase 1 验收（确定性 MVP）

### 验收目标

- 在 Android+iOS 模拟器上稳定跑通核心登录主链路。

### 你当前本地环境可直接使用

- Android Studio Emulator（建议固定 1 个 API Level 做基线）
- Xcode Simulator（建议固定 1 个 iPhone 机型 + iOS 版本）

### 核心指标（建议初始门槛）

- 主链路通过率：>= 95%（样例/demo app 每平台至少 5 次连续执行）
- 失败证据包完整率：>= 99%
- 中位动作时延：满足 roadmap 中定义目标
- OCR/CV 参与动作比例：保持低占比并可追踪
- 已知系统弹窗/底部弹窗命中时，能够被检测并按规则处理

对于复杂度更高的真实业务 App，可将 5 次提升到 10 次或更高，但不建议对当前最小 RN 登录样例使用过高轮次。

### 验收用例（最小集合）

1. 启动 App -> 到登录页
2. 输入手机号/密码 -> 登录
3. 验证首页关键元素可见
4. 失败时自动输出完整证据包
5. 若出现已知系统弹窗（如存储密码），能够记录并处理后继续流程

### 证据要求

- 每次 run 的 action timeline
- 关键节点截图 + tree dump
- app/device logs
- 失败分类统计
- interruption 事件记录（命中类型、处理动作、是否恢复成功）

### Phase 1 完成判定

如果以下条件同时满足，则可以认为 Phase 1 已完成：

1. iOS 黄金登录链路通过率达标
2. Android 黄金登录链路通过率达标
3. 已知系统中断（如存储密码弹窗）可检测、可记录、可处理
4. 失败证据包完整率达标
5. 没有阻断黄金链路的 P0 问题

满足以上条件后，不需要继续额外拉长样例项目的重复次数，可以进入 Phase 2。

---

## 4. Phase 2 验收（稳定性与兜底）

### 验收目标

- 对动画、弹窗、弱网、偶发状态漂移具备稳定处理能力。

### 指标建议

- 相比 Phase 1，Flaky 比例显著下降（目标值在 phase charter 中明确）
- 回退链路可观测率 100%（每次 fallback 都有 trace）
- 低置信度 OCR/CV 不可直接放行动作（策略校验 100%）
- interruption policy library 覆盖目标提示类型，且自动处理成功率达到阶段阈值

### Phase 2 完成后是否进入 Phase 3

可以，但前提是：

1. 执行核心已经稳定，不会因为新增框架支持而频繁被基础问题拖垮
2. interruption / fallback / retry / evidence 这些底层能力已可复用
3. 已知 P0/P1 稳定性问题有明确结论（已修复或已接受）

满足后即可进入 Phase 3（RN/Flutter 扩展）。

---

## 5. Phase 3 验收（RN/Flutter 扩展）

### 验收目标

- 在 Native 之外，至少 1 个框架 profile（RN 或 Flutter）达到可用门槛。

### 指标建议

- 框架 onboarding checklist 完成率 100%
- 兼容矩阵已实例化并可追踪
- 核心链路稳定性达到阶段阈值

---

## 6. Phase 4 验收（治理与企业化）

### 验收目标

- 权限策略、审计、脱敏、留存策略可落地执行。

### 指标建议

- 策略拦截准确率 100%
- 审计追踪完整率 100%
- 脱敏策略抽检通过率 100%

---

## 7. Phase 5 验收（智能化增强）

### 验收目标

- 自愈建议与智能规划能力可控、可解释、可回滚。

### 指标建议

- 自愈建议采纳率与回归通过率可量化
- 智能建议均可追溯来源与变更影响

---

## 8. 模拟器与真机的验收分层建议

## L1（本地模拟器，当前即可执行）

- 验证主功能、主链路稳定性
- 验证协议、日志、证据包

## L2（CI 模拟器集群）

- 验证并发和回归稳定性
- 统计波动指标

## L3（小规模真机冒烟）

- 验证系统弹窗、通知、厂商差异、性能边界

> 结论：你现在有 Xcode + Android Studio 模拟器，完全可以作为 **Phase 1 主验收环境**。

---

## 9. Go / No-Go 决策规则（建议）

满足以下条件才进入下一阶段：

1. 阶段指标全部达标
2. 证据包完整且可复核
3. 风险项有明确 owner 和处理计划
4. Review Checklist 全通过

否则保持 No-Go，修复后复验。

---

## 10. 发现新问题时，如何系统补回方案

像这次“存储密码弹窗”这种情况，不应该只修当前 flow，而应该执行固定流程：

1. **先留证据**：截图、tree、日志、动作时间线
2. **归类问题**：它属于能力缺口、平台适配缺口，还是验收遗漏
3. **补文档**：至少更新能力图、相关架构/适配器文档、验收标准
4. **补规则/实现**：增加 interruption rule / policy / flow 兜底
5. **从头复验受影响链路**：不能只从中间步骤继续
6. **记录结论**：确认它属于哪个 phase、是否影响当前 Go/No-Go

建议把这种机制视为项目常规流程，而不是异常流程。
